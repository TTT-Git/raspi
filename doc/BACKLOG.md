# 改修バックログ

## この文書の目的

Raspberry Piで動作する室温監視・エアコン制御システムについて、完了済みの変更と今後の課題を優先度順に管理する。
未実装項目は、背景、目的、方針、完了条件を確認してから一項目ずつ実装する。

状態は次のいずれかで管理する。

- 完了
- 議論中
- 未着手
- 保留

## Done

### DB-001: SQLiteロックで停止しない

- **ID:** DB-001
- **状態:** 完了
- **優先度:** Done

#### 背景

DBバックアップや複数処理の競合時に`database is locked`が発生し、センサー取得プロセスやエアコン制御スレッドが終了していた。

#### 実施内容

- SQLite接続タイムアウト、`busy_timeout`、WALを設定した。
- DBロック時の回数制限付きリトライを実装した。
- commit失敗時のrollbackとセッション解放を実装した。
- リトライ上限後はその周期だけ処理をスキップし、次周期へ進むようにした。
- センサー取得ループとエアコン制御ループがDB例外で終了しないようにした。

#### 完了確認

- 実装済み
- 一時SQLite DBによるテスト済み
- `main`反映済み
- Raspberry Pi実機で運用中

#### 注意点

`database_locked_retry`や`database_locked_skipped`は、DBロック中も処理を継続するためのログであり、プロセスの異常終了を意味しない。

- 停止完了後に再開始できるようにした。

#### 完了確認

- Raspberry Pi実機確認済み
- `main`反映済み

#### 注意点

## Next / P0

### CTRL-001: 温度制御を安定化する

- **ID:** CTRL-001
- **状態:** 議論中
- **優先度:** P0

#### 背景

実機ログで、冷房設定温度が短時間に大きく変化している。

例:

```text
23 -> 24 -> 26 -> 27 -> 29 -> 30 -> 31
```

現在の制御は、温度の短期的な変化や予測値に反応し、設定温度を頻繁または大幅に変更する可能性がある。
暖房と冷房の切り替えについても、同じ安定化設計の中で慎重に扱う必要がある。

#### 目的

- 設定温度の変更を滑らかにする。
- 短時間の過剰な制御変動を抑える。
- 室温を許容範囲内に保ちながら、IR送信と設定変更の回数を減らす。
- 暖房と冷房の不要な切り替えを抑える。

#### 方針案

- 1周期あたりの設定温度変更量を最大1℃に制限する。
- 設定温度の最小変更間隔を設ける。
- 直近データの移動平均または中央値を判断値として使う。
- 目標温度帯にヒステリシスとデッドバンドを設ける。
- 最小運転時間と最小停止時間を設ける。
- 冷房・暖房モードの最小保持時間と切替禁止時間を設ける。
- 反対モードの条件が複数周期継続した場合だけ切り替える。
- 制御判断、変更前後の設定温度、変更理由をログへ残す。

#### 完了条件

- 短時間で設定温度が2℃以上飛ばない。
- 設定温度の変更間隔が定義した下限を下回らない。
- 制御ログから設定変更の理由を追跡できる。
- 単体テストで1周期あたりの変更量上限を確認できる。
- 過去データまたはIR送信なしのテストで、現行方式より設定変更回数が減る。
- 冷房と暖房が短時間に直接往復しない。

#### 注意点

- 目標室温、許容温度範囲、平均化方法、最小変更間隔は実装前に決定する。
- 実機テスト前に、IR送信を行わない単体テストまたはリプレイテストを実施する。
- 温度制御ロジックとモード切替ロジックを一度に大きく変更せず、段階的に検証する。

### OPS-001: 長期運用整備

- **ID:** OPS-001
- **状態:** 未着手
- **優先度:** P1

#### 背景

現在は手動または`nohup`で起動しており、Raspberry Pi再起動後やプロセス異常終了後の復旧が自動化されていない。

#### 目的

- SSH切断に依存せず、安定してプロセスを運用する。
- Raspberry Pi再起動後や異常終了後に、決められた方法で復旧できるようにする。
- センサー取得とWebサーバーの起動順序を明確にする。

#### 方針案

- `test_veiws.py`と`test.py`をsystemdサービスとして管理する。
- サービスを分離するか、一つにまとめるかを決める。
- `Restart`、再起動間隔、実行ユーザー、作業ディレクトリを定義する。
- センサー取得とWebサーバーの起動順序、依存関係を定義する。
- 自動復旧時にエアコン制御を自動開始するか決める。

#### 完了条件

- Raspberry Pi再起動後に必要なプロセスが起動する。
- プロセス異常終了後に定義した条件で再起動する。
- 同じサービスが二重起動しない。
- `systemctl`で起動、停止、状態確認ができる。

#### 注意点

`test.py`起動直後は制御停止という現在の安全側の仕様を維持する。

### LOG-001: ログローテーション

- **ID:** LOG-001
- **状態:** 未着手
- **優先度:** P1

#### 背景

`aircon_ai.log`や`log/*.log`が長期運用で肥大化し、ストレージを圧迫する可能性がある。

#### 目的

必要な期間のログを保持しながら、ログファイルの無制限な肥大化を防ぐ。

#### 方針案

- Python loggingの`RotatingFileHandler`または`TimedRotatingFileHandler`を検討する。
- OS側の`logrotate`を使用する案と比較する。
- systemd化する場合は`journald`へ集約する案も比較する。
- 保持日数、ファイルサイズ、世代数を決める。

#### 完了条件

- ログの最大サイズまたは保持期間が制限されている。
- ローテーション後もプロセスがログを書き続けられる。
- ログ確認と削除の運用手順が文書化されている。

#### 注意点

`OPS-001`のログ出力先とサービス構成を決めてから最終方式を選ぶ。

### DB-002: WAL対応バックアップ手順のスクリプト化

- **ID:** DB-002
- **状態:** 未着手
- **優先度:** P1

#### 背景

WAL有効時は`temp_humid.sql-wal`と`temp_humid.sql-shm`が存在する可能性があり、稼働中に`temp_humid.sql`本体だけをコピーすると不完全なバックアップになる可能性がある。

#### 目的

稼働中のDBを破損や欠落なく、安全にスナップショットとして保存する。

#### 方針案

- SQLite Backup APIを使う専用スクリプトを作成する。
- SQLite CLIの`.backup`を利用する方法と比較する。
- バックアップ先、ファイル名、保存世代数、実行頻度を決める。
- バックアップ中もセンサー取得とWebサーバーを停止させない。

#### 完了条件

- WAL使用中に整合性のあるバックアップを作成できる。
- バックアップ中も本番プロセスが継続する。
- バックアップDBを読み取り専用で開き、最新データを確認できる。
- 復元手順が文書化されている。

#### 注意点

本番DBを使った検証では、書き込みや長時間ロックを発生させない。

## Deferred / P2

### SAFE-001: 古い温度データを使わない

- **ID:** SAFE-001
- **状態:** 保留
- **優先度:** P2

#### 背景

- `raspi0-1.local`の名前解決に失敗している。
- 古い`2024-02-05`のデータがlatest扱いで表示されている。
- 一方、現在の制御に使用しているraspi4B自身の温度・湿度・CO2は取得できている。

#### 目的

将来的に、古い温度データや取得時刻が不明なデータを現在値として制御へ使用しないようにする。

#### 方針案

- 温度データにfreshness thresholdを設ける。
- 閾値を超えたデータは`stale_data`として扱う。
- 制御対象に使う温度データが古い場合はIR送信しない。
- 状態APIへ最終取得時刻、経過時間、制御抑止理由を返す。

#### 完了条件

- 制御対象データが設定時間以上古い場合、IR送信されない。
- 古いデータであることをログと状態APIから確認できる。
- 新しいデータ取得後の復帰条件が定義されている。

#### 注意点

raspi4B自身の制御用温度を取得できている間は、今すぐの最優先課題とはしない。
保留解除時に、制御対象の識別方法と鮮度閾値を再確認する。

### DATA-001: `raspi0-1.local`の名前解決失敗を整理する

- **ID:** DATA-001
- **状態:** 保留
- **優先度:** P2

#### 背景

`raspi0-1.local`の名前解決または接続に失敗し、古いデータが残っている。

#### 目的

不要な監視対象と必要な監視対象を整理し、接続失敗ログと古い表示データを減らす。

#### 方針案

- 不要な端末であれば監視対象から外す。
- 必要な端末であればhostname、固定IP、mDNS、Avahi設定を確認する。
- 接続失敗ログの出力頻度を制限する。
- UI上で到達不能または古いデータと分かる表示を検討する。

#### 完了条件

- `raspi0-1.local`を監視対象として残すかどうか決定している。
- 残す場合は安定して名前解決またはIP接続できる。
- 外す場合は不要な取得処理と接続失敗ログが発生しない。

#### 注意点

raspi4Bの制御用センサーデータ取得とは分離して扱う。

## 次に着手すべき候補

次は`CTRL-001`を第一候補とする。
実機ログで設定温度が短時間に大きく変化しており、現在の利用品質へ直接影響しているためである。

`CTRL-001`の実装前に、少なくとも次を決める。

1. 目標室温と許容温度範囲
2. 1周期あたりの最大変更量
3. 設定温度の最小変更間隔
4. 移動平均または中央値の対象時間
5. 冷房・暖房の固定、自動切替、切替禁止時間
6. IR送信なしで比較するための評価データと評価指標
