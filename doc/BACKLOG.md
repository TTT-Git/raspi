# 改修バックログ

## この文書の目的

この文書は、室温監視・エアコン制御システムの改修候補を優先度順に管理するためのバックログである。
各項目は、設計を議論して合意した後に実装へ進める。優先度が同じ項目についても、依存関係と実機への影響を確認して着手順を決める。

状態は次のいずれかで管理する。

- 未着手
- 議論中
- 実装候補
- 実装中
- 完了

## 優先度一覧

| ID | 優先度 | 状態 | 概要 |
| --- | --- | --- | --- |
| DB-001 | P0 | 議論中 | SQLiteのロックでプロセスを停止させない |
| WEB-001 | P0 | 議論中 | ブラウザの「開始」ボタンから確実に制御を開始する |
| SAFE-001 | P1 | 未着手 | 古い温度データによるIR制御を禁止する |
| OPS-001 | P1 | 未着手 | 長期運用と障害復旧の手順を整備する |
| CTRL-001 | P2 | 議論中 | 室温を安定させる制御方式を設計する |
| CTRL-002 | P2 | 議論中 | 暖房と冷房の頻繁な切り替えを抑制する |
| UI-001 | P2 | 未着手 | 制御状態と制御抑止理由をWeb UIに表示する |
| DB-002 | P3 | 未着手 | DB肥大化への対策を設計する |

## DB-001: SQLiteのロックでプロセスを停止させない

- **ID:** DB-001
- **優先度:** P0
- **状態:** 議論中

### 背景

センサーデータ収集処理、エアコン制御処理、Web APIが同じSQLite DBを読み書きする。
書き込み競合や長時間の読み込みが発生すると、`SQLite database is locked`により処理が例外終了する可能性がある。
特に、センサーデータ収集処理やエアコン制御スレッドが終了すると、監視や制御が無言で停止する危険がある。

### 目的

一時的なDBロックやコミット失敗が発生しても、プロセスと制御ループを終了させず、安全にリトライまたは制御抑止へ移行させる。

### 対象ファイル候補

- `models/base.py`
- `test_veiws.py`
- `controllers/streamdata.py`
- `controllers/aircon_stream.py`
- `models/ai_aircon_ctrl.py`
- `settings.ini`
- `settings.py`

### 実装方針案

- SQLiteをWALモードで運用する。
- 接続時に`busy_timeout`を設定する。
- DB書き込みを回数制限付きでリトライする。
- コミット失敗時は必ずrollbackし、セッションを破棄または正常化する。
- ロック取得中に例外が発生しても、Python側のロックを確実に解放する。
- `OperationalError`などDB関連例外を制御ループの外まで伝播させない。
- センサー収集ループとエアコン制御ループの最上位に例外処理を置き、失敗を記録して次周期へ進む。
- リトライ回数を超えた場合は、IR送信を行わずエラー状態として公開する。

### 議論したいこと

- WAL設定をアプリ起動時に行うか、DB初期化・運用手順として行うか。
- `busy_timeout`の値を何秒にするか。
- 書き込みリトライの回数、間隔、バックオフ方式。
- 読み込み失敗時もリトライするか、直ちに制御を抑止するか。
- 現在のプロセス内`threading.Lock`を維持するか、SQLite側の排他制御を中心にするか。
- DB障害中のWeb API応答形式と状態表示。

### 完了条件

- 一時的なDBロックを再現しても、センサー収集プロセスとエアコン制御スレッドが終了しない。
- コミット失敗後にrollbackされ、次回のDB操作が継続できる。
- リトライ上限を超えた場合はIR送信されず、エラー理由がログと状態APIに残る。
- WAL、タイムアウト、リトライの設定値と確認方法が文書化されている。

## WEB-001: ブラウザの「開始」ボタンから確実に制御を開始する

- **ID:** WEB-001
- **優先度:** P0
- **状態:** 議論中

### 背景

Web UIの開始ボタンは`POST /api/aircon/start/`へ接続されているが、`test.py`が別経路で制御スレッドを直接開始している。
そのスレッドは`AirconStream.thread`で管理されないため、APIから見た状態と実際の制御ループが一致しない可能性がある。
停止中のスレッドがスリープしている間に再開始すると、多重起動する可能性もある。

### 目的

制御ループの所有者とライフサイクルを一か所に集約し、ブラウザから開始・停止した結果を正確に確認できるようにする。

### 対象ファイル候補

- `test.py`
- `controllers/aircon_stream.py`
- `controllers/webserver.py`
- `views/jquery.html`

### 実装方針案

- 制御スレッドの生成、保持、停止、再開を`AirconStream`に統一する。
- `test.py`から`stream_aircon_ctrl`を直接Threadへ渡す処理を廃止または明示的な自動開始APIへ置き換える。
- 開始・停止処理をロックで排他制御する。
- `STOPPED / STARTING / RUNNING / STOPPING / ERROR`の状態を管理する。
- `sleep()`の代わりに停止要求へ即時反応できる`threading.Event.wait()`を検討する。
- 同一プロセス内で制御スレッドが1本だけ存在することを保証する。
- 開始APIは状態遷移が確定した後の状態を返す。

### 議論したいこと

- アプリ起動時は停止状態にするか、自動開始するか。
- 「開始」の意味を制御ループ開始とするか、直ちにエアコンONとするか。
- 停止時に必ずエアコンOFFのIRを送信するか。
- スレッドが異常終了した場合に自動再起動するか、ユーザー操作を待つか。
- 複数プロセスでWebアプリが起動された場合の排他方法。

### 完了条件

- 開始ボタンを1回押すと、制御スレッドが1本だけ開始する。
- 連打や複数リクエストでも多重起動しない。
- 停止後に再開始でき、古いスレッドが復活しない。
- APIとWeb UIの状態表示が実際のスレッド状態と一致する。
- 異常終了時に`ERROR`と理由を確認できる。

## SAFE-001: 古い温度データによるエアコン制御を禁止する

- **ID:** SAFE-001
- **優先度:** P1
- **状態:** 未着手

### 背景

センサー取得処理が停止しても、DBには最後に取得した値が残る。
古い温度を現在温度として扱うと、実際の室温と異なるIR操作を行う危険がある。

### 目的

温度データの鮮度とDB取得結果を制御実行の前提条件とし、安全を確認できない場合はIR送信を禁止する。

### 対象ファイル候補

- `models/ai_aircon_ctrl.py`
- `models/base.py`
- `controllers/aircon_stream.py`
- `controllers/webserver.py`
- `views/jquery.html`
- `settings.ini`
- `settings.py`

### 実装方針案

- 最新温度時刻と現在時刻の差を毎周期確認する。
- 5分以上古い場合は制御判断とIR送信を行わない。
- DB読み込み失敗、対象テーブル不明、温度値欠損の場合も制御しない。
- 状態APIへ`latest_temperature_time`、`data_age_seconds`、`inhibit_reason`を返す。
- Web UIに「温度データが古いため制御停止中」などの警告を表示する。
- 鮮度閾値を設定値として管理する。

### 議論したいこと

- 5分という閾値が、通常の取得周期と障害検知時間として妥当か。
- データ復旧後に自動で制御を再開するか、手動確認を必要とするか。
- 古いデータ検出時にエアコンOFFを送るか、IR送信を一切行わないか。
- 時刻のタイムゾーンと時計ずれをどう扱うか。

### 完了条件

- 最新温度データが5分以上古い場合、IR送信が行われない。
- DB読み込み失敗時に制御スレッドが終了しない。
- 状態APIとWeb UIで、最終温度時刻と抑止理由を確認できる。
- 新しいデータが到着した場合の復帰動作が仕様化されている。

## OPS-001: 長期運用と障害復旧の手順を整備する

- **ID:** OPS-001
- **優先度:** P1
- **状態:** 未着手

### 背景

現在の起動方法、監視方法、停止時の復旧方法がリポジトリ内で明確に定義されていない。
手動起動や`nohup`だけでは、異常終了後の自動復旧、ログ確認、多重起動防止が不十分になる可能性がある。

### 目的

Raspberry Pi再起動後やプロセス異常終了後も、決められた手順で安全に起動・確認・復旧できる運用を確立する。

### 対象ファイル候補

- 新規の運用手順書
- systemd service定義候補
- `test.py`
- `test_veiws.py`
- ログ設定を持つモジュール

### 実装方針案

- 当面の`nohup`運用と、最終的なsystemd運用を比較する。
- Web、センサー収集、エアコン制御を同一サービスにするか分離する。
- systemdの`Restart`、`WorkingDirectory`、実行ユーザー、環境変数を明示する。
- PIDまたはサービス管理により多重起動を防止する。
- `journalctl`とファイルログの確認方法を統一する。
- 停止、再起動、障害確認、復旧の手順書を作成する。

### 議論したいこと

- `nohup`を暫定運用として残すか、最初からsystemdへ移行するか。
- 各処理を別サービスに分割するか。
- 異常終了時の自動再起動回数と間隔。
- 自動再起動時にエアコン制御を自動開始してよいか。

### 完了条件

- 標準の起動・停止・状態確認コマンドが決まっている。
- 同じサービスを重複起動できない。
- 異常終了を検知し、決められた方法で復旧できる。
- ログの保存先、確認方法、代表的な障害の切り分け手順が文書化されている。

## CTRL-001: 室温を安定させる制御方式を設計する

- **ID:** CTRL-001
- **優先度:** P2
- **状態:** 議論中

### 背景

現在は目標温度との差が小さくても、判定範囲を外れると最低1℃ずつ設定温度を変更する。
短期間の予測温度変動にも反応するため、設定温度が頻繁に変化しやすい。

### 目的

室温の変動、IR送信回数、エアコン設定変更回数を抑えながら、合意した快適温度範囲を維持する。

### 対象ファイル候補

- `models/ai_aircon_ctrl.py`
- `controllers/aircon_stream.py`
- `settings.ini`
- `settings.py`
- 制御ロジックのテストファイル候補

### 実装方針案

- ON判定とOFF判定に異なる閾値を使うヒステリシスを導入する。
- 目標付近に操作しないデッドバンドを設ける。
- 直近5～10分の移動平均または中央値を判断値に使う。
- 最小運転時間と最小停止時間を設ける。
- 設定温度変更を一定時間に1回までに制限する。
- 1回の設定温度変更量を最大1℃などに制限する。
- 目標温度との差に応じて、維持・弱い補正・強い補正の段階制御を行う。

### 議論したいこと

- 目標室温と許容温度範囲。
- 平均と中央値のどちらを採用するか。
- 最小運転時間、最小停止時間、設定変更間隔。
- 室温安定性、消費電力、設定変更回数のどれを優先するか。
- 予測温度を制御の主判定に残すか、補助情報へ変更するか。

### 完了条件

- 制御方式と全閾値が文書化されている。
- 過去データによるリプレイで、現行方式より設定変更回数が減る。
- 許容温度範囲内の滞在率など、評価指標が定義されている。
- 実機適用前にIR送信なしの検証が完了している。

## CTRL-002: 暖房と冷房の頻繁な切り替えを抑制する

- **ID:** CTRL-002
- **優先度:** P2
- **状態:** 議論中

### 背景

現在は、冷房設定が上限、または暖房設定が下限へ到達した後、予測温度が狭い判定範囲を外れると反対モードへ切り替わる。
室温の遅れや予測値の揺れにより、冷房と暖房が短時間で往復する可能性がある。

### 目的

不要なモード切り替えを防ぎ、設備負荷、消費電力、室温の振動を抑える。

### 対象ファイル候補

- `models/ai_aircon_ctrl.py`
- `controllers/aircon_stream.py`
- `controllers/webserver.py`
- `views/jquery.html`
- `settings.ini`
- `settings.py`

### 実装方針案

- `冷房固定 / 暖房固定 / 自動`を運転モードとして明示する。
- 一度選択したモードを維持する最小モード保持時間を設ける。
- 目標温度周辺に、冷房も暖房も開始しない中立ゾーンを設ける。
- 暖房から冷房、冷房から暖房への切り替え後に切替禁止時間を設ける。
- 反対モードの条件が複数周期連続して成立した場合だけ切り替える。
- モード切り替え時は`CHANGEOVER_WAIT`を経由し、直接切り替えない。

### 議論したいこと

- 通常運用を固定モード中心にするか、自動モード中心にするか。
- 暖房と冷房の自動切り替えを許可するか。
- 最小モード保持時間と切替禁止時間。
- 中立ゾーンの幅。
- 外気温や季節設定を切替判断に利用するか。

### 完了条件

- 運転モードと切替条件が状態遷移図として定義されている。
- 冷房と暖房が直接切り替わらない。
- 切替禁止時間中は反対モードのIRが送信されない。
- 過去データによる検証で短時間の往復が発生しない。

## UI-001: 制御状態と制御抑止理由をWeb UIに表示する

- **ID:** UI-001
- **優先度:** P2
- **状態:** 未着手

### 背景

現在のUIは主に実行中・停止中を表示するが、制御スレッドのエラー、古い温度、DBロック、IR未送信理由を確認できない。
開始操作が成功したように見えても、実際には制御が抑止されている可能性がある。

### 目的

ユーザーがブラウザだけで、制御状態、使用中の温度データ、異常、直近の制御判断を確認できるようにする。

### 対象ファイル候補

- `controllers/webserver.py`
- `controllers/aircon_stream.py`
- `models/ai_aircon_ctrl.py`
- `views/jquery.html`

### 実装方針案

- 状態APIに`STOPPED / STARTING / RUNNING / STOPPING / ERROR`を返す。
- 最新温度時刻とデータ経過秒数を返す。
- DBロックの発生回数または直近発生時刻を返す。
- 直近のIRコマンド、送信時刻、成功・失敗を返す。
- IRを送信しなかった場合は、デッドバンド、古いデータ、最小変更間隔などの理由を返す。
- 警告とエラーを通常状態と区別して表示する。

### 議論したいこと

- 状態情報をメモリだけで保持するか、DBへ保存するか。
- DBロック件数を累積値にするか、直近イベントだけにするか。
- UIに表示する情報量と更新間隔。
- IR送信成功をSSHコマンド終了だけで判定してよいか。

### 完了条件

- 実行中、停止中、開始中、停止中、エラーを区別できる。
- 最新温度時刻とデータ鮮度を確認できる。
- 制御しなかった理由を確認できる。
- DBロックとIR送信失敗が画面に警告表示される。

## DB-002: DB肥大化への対策を設計する

- **ID:** DB-002
- **優先度:** P3
- **状態:** 未着手

### 背景

温湿度データとエアコン状態を継続的に同一DBへ保存しているため、長期運用でDBが肥大化する。
本番DBを直接コピーして分析すると、コピー時間、ロック、ストレージ消費へ影響する可能性がある。

### 目的

本番の収集・制御を止めずに、必要な期間のデータ保持、バックアップ、分析、リプレイテストを行える運用を確立する。

### 対象ファイル候補

- `models/base.py`
- DB保守スクリプト候補
- バックアップ・アーカイブ手順書
- リプレイテスト用ツール候補
- `settings.ini`
- `settings.py`

### 実装方針案

- 保持期間を決め、古いデータを別DBへアーカイブする。
- 本番DBには直近数か月など必要な範囲だけ残す。
- SQLiteのオンラインバックアップAPIまたはWAL対応の安全なバックアップ方式を採用する。
- リプレイテスト用DBを本番DBから分離する。
- アーカイブ後の`VACUUM`実行条件と停止時間を運用手順として定義する。

### 議論したいこと

- 本番DBに保持する期間。
- 生データを永続保存する必要があるか。
- 古いデータを間引いて集計値だけ残すか。
- バックアップ先、頻度、世代数。
- リプレイテスト用データの匿名化や切り出し方法。

### 完了条件

- 保持期間、アーカイブ、削除、バックアップの方針が決まっている。
- 本番処理を停止せず、整合性のあるバックアップを取得できる。
- リプレイテストが本番DBを直接変更せず実行できる。
- 復元手順が文書化され、検証されている。

## 推奨する議論順序

1. `DB-001`: プロセスが停止しないDBアクセス方式
2. `WEB-001`: 制御スレッドの所有権と開始・停止状態
3. `SAFE-001`: 古いデータとDB障害時の制御抑止
4. `OPS-001`: 起動、監視、復旧方法
5. `CTRL-001`と`CTRL-002`: 温度制御とモード切替
6. `UI-001`: 決定した状態・抑止理由の表示
7. `DB-002`: 保持期間とアーカイブ運用
