### CTRL-001: 2段階冷房制御

- **ID:** CTRL-001
- **状態:** 完了
- **優先度:** Done

#### 背景

実機ログでは室温が比較的安定している一方で、冷房設定温度が短時間に大きく変化していた。

例:

```text
23 → 24 → 26 → 27 → 29 → 30 → 31
```

今回のログとDBの解析では、3分先の線形予測は、現在温度をそのまま判断に使う方式より有利ではなかった。
既存制御は、次の要因により温度の短期変動へ過敏に反応している。

- 許容温度幅が狭い。
- 直近3分の温度傾きを、さらに3分先まで外挿している。
- `ceil(gap_temp)`により、小さな温度差でも設定温度が1℃以上変更される。
- 設定温度の最小変更間隔がない。
- 条件が複数周期継続したことを確認する連続判定がない。

ただし、エアコンを停止して外出し、帰宅時には室温を早く下げたいという要件がある。
通常時の変更を常に最大1℃、15分間隔に制限するだけでは、開始直後の急速冷却が遅くなる可能性がある。
そのため、開始直後の急速復帰と通常時の安定維持を分離する。

#### 目的

- 帰宅時など室温が高い場合は、開始直後に十分な冷房能力を確保する。
- 目標温度へ近づいた後は、設定温度とIR送信の過剰な変動を抑える。
- 急速復帰から安定維持への移行条件を明確にする。
- 制御判断と設定変更の理由をログから追跡可能にする。

#### 採用方針

- 冷房・暖房は当面手動固定とする。
- 自動冷暖房切替は無効にする。
- 3分先予測は主判定から外し、比較・記録用のログに限定する。
- 冷房制御を`RECOVERY_COOLING`と`STABLE_COOLING`の2段階に分ける。

#### 制御状態

```text
STOPPED
  ↓ 冷房制御開始
RECOVERY_COOLING
  ↓ 急速復帰終了条件
STABLE_COOLING
  ↓ 停止操作
STOPPED
```

開始時に急速復帰条件を満たさない場合は、`STOPPED`から直接`STABLE_COOLING`へ移行する。

#### RECOVERY_COOLING: 急速復帰

##### 開始条件

- 冷房制御開始時に、直近5分の温度中央値が`目標温度 + 1.3℃`以上である。
- 目標温度25.7℃の場合は、中央値が27.0℃以上である。
- 将来の手動急速冷房に備え、判定入力の`force_recovery`でも急速復帰を開始できる設計とする。ただしAPI/UI接続は別タスクとする。

##### 動作

- 冷房モードに固定する。
- 開始時に冷房設定温度を一度だけ24℃へ変更する。
- 風量変更を安全に扱える場合は、`auto`または強めの風量を検討する。
- 急速復帰中は3分ごとに設定温度を変更せず、24℃を維持する。
- 暖房への自動切替は行わない。

##### 終了条件

次のいずれかを満たした場合、急速復帰を終了する。

- 直近5分の温度中央値が`目標温度 + 0.5℃`以下になる。
- 目標温度25.7℃の場合は、中央値が26.2℃以下になる。
- 急速復帰開始から45分が経過する。

##### 終了時

- 冷房設定温度を26℃へ戻す。
- 状態を`STABLE_COOLING`へ移行する。

#### STABLE_COOLING: 安定維持

##### 制御設定

| 項目 | 初期案 |
| --- | --- |
| 判断温度 | 直近5分の温度中央値 |
| 目標温度 | 25.7℃ |
| デッドバンド | 25.3〜26.1℃ |
| 連続判定 | 2周期 |
| 最小変更間隔 | 15分 |
| 1回の最大変更量 | 1℃ |
| 3分先予測 | ログ記録のみ |

##### 冷房固定時の制御

- 中央値が26.1℃を超える状態が2周期継続した場合、冷房設定温度を1℃下げる。
- 中央値が25.3℃を下回る状態が2周期継続した場合、冷房設定温度を1℃上げる。
- 中央値が25.3〜26.1℃の場合、設定温度を変更しない。
- 前回の設定変更から15分未満の場合、条件を満たしても設定温度を変更しない。

#### 制御ログ

各制御周期で少なくとも次を記録する。

- 現在の制御状態
- 判断に使用した5分中央値
- デッドバンドの判定結果
- 同方向条件の連続判定回数
- 最終設定変更時刻
- 変更前後の設定温度
- 設定を変更した、または変更しなかった理由
- 3分先予測温度。ただし制御判断には使用しない。

#### 実施内容

- 副作用を持たないpure decision logicを実装した。
- 過去DBと固定サンプルを使った時系列リプレイ検証を実施した。
- 急速復帰の開始条件を`目標温度 + 1.5℃`から`目標温度 + 1.3℃`へ調整した。
- `AirconStream`へ`two_stage_dry_run`を接続した。
- `RealCoolingCommandAdapter`と既存`Aircon`を呼び出すsender wrapperを実装した。
- `AIRCON_CONTROL_MODE=two_stage_real`を明示した場合だけ実IR制御を使用するようにした。
- 設定未指定時と`settings.ini`のデフォルトは`legacy`のままとした。
- IR送信成功後だけ制御状態と設定温度を確定するようにした。
- `two_stage_real`ではIR成功後に`AirconState.create()`を呼び、既存Webグラフ用の冷房設定温度を記録するようにした。
- `two_stage_dry_run`ではIR送信と`AirconState`記録を行わない。
- legacy制御は既存の`Ai.ctrl_temp()`を維持し、two-stage制御と同一周期に二重実行しないようにした。

#### リプレイ確認

- pure decision logicとRunner/Adapterの単体テストを実施した。
- 急速復帰、安定維持、デッドバンド、連続判定、15分の最小変更間隔を確認した。
- `STABLE_COOLING`で1回の設定温度変更量が1℃以内であることを確認した。
- デッドバンド内で設定温度を変更しないことを確認した。

#### Raspberry Pi実機確認

- `two_stage_dry_run`で判定結果だけが記録され、実IRが送信されないことを確認した。
- `two_stage_real`で実IR制御が動作することを確認した。
- 開始時に`ensure_cooling`で冷房26℃が送信されることを確認した。
- 低温側条件の確認後、`set_cooler_temp`で27℃、28℃へ1℃ずつ変更されることを確認した。
- その後、温度がデッドバンド内に入ると`within_deadband`、`noop`が継続することを確認した。
- `AirconState.create()`による記録後、Web画面のエアコン設定温度グラフが更新されることを確認した。
- 実機確認中に、短時間で設定温度が連続的に大きく変化する現象は発生しなかった。

#### 完了確認

- 通常維持中に、設定温度が短時間で大きく飛ばない。
- 通常維持中の設定変更回数を2回/時以下にすることを評価目標にできる。
- `STABLE_COOLING`で1回の設定温度変更量が1℃を超えない。
- 急速復帰時は開始直後に24℃を設定し、十分な冷房能力を確保できる。
- 温度または時間条件により、急速復帰から安定維持へ移行できる。
- 過去DBを使ったリプレイテストで、現行方式より設定変更回数が減る。
- 制御ログから、状態、判断温度、連続判定、最終変更時刻、変更理由を確認できる。
- IR送信をモックした単体テストで、各状態遷移と設定温度変更を確認できる。
- Raspberry Piでdry-runとreal制御を確認済み。
- Webグラフへの設定温度反映を確認済み。
- featureブランチ上で実装・実機確認済み。`main`へのマージは未実施。

#### 実装履歴

1. 温度判断処理とIR送信処理を分離する。
2. 直近5分の温度中央値を計算する。
3. `RECOVERY_COOLING`と`STABLE_COOLING`の状態を追加する。
4. 急速復帰の開始条件、終了温度、45分の終了条件を実装する。
5. 安定維持のデッドバンド、2周期連続判定、15分制限、最大1℃変更を実装する。
6. 3分先予測を制御判断から外し、ログ記録だけにする。
7. 過去DBを使ったリプレイテストを作成する。
8. 冷房固定モードでRaspberry Pi実機確認を行う。

#### 注意点

- 最初の実装対象は冷房固定に限定する。
- 暖房制御と冷暖房の自動切替は`CTRL-001`の対象外とする。
- 24℃、26℃、1.3℃、0.5℃、45分、15分は初期値であり、リプレイと実機ログを基に再評価する。
- 風量変更は既存IR実装と実機の安全性を確認できた場合だけ導入する。
- 実機確認前に、IR送信を行わない単体テストとリプレイテストを実施する。
- 自動切替を追加する場合は別タスクとし、最小モード保持2時間、目標±1.0℃以上、5周期連続、OFF待機15〜30分などを検討する。
- 実機ログに`Unable to set line 5 to input`が出る場合がある。現時点ではIR送信や制御の失敗にはなっていないため、GPIO初期化・解放処理の確認を別タスク候補とする。

## Deferred / P2

### SAFE-001: 古い温度データを使わない

- **ID:** SAFE-001
- **状態:** 保留
- **優先度:** P2

#### 背景

- `raspi0-1.local`の名前解決に失敗している。
- 古い`2024-02-05`のデータがlatest扱いで表示されている。
- 一方、現在の制御に使用しているraspi4B自身の温度・湿度・CO2は取得できている。

#### 目的

将来的に、古い温度データや取得時刻が不明なデータを現在値として制御へ使用しないようにする。

#### 方針案

- 温度データにfreshness thresholdを設ける。
- 閾値を超えたデータは`stale_data`として扱う。
- 制御対象に使う温度データが古い場合はIR送信しない。
- 状態APIへ最終取得時刻、経過時間、制御抑止理由を返す。

#### 完了条件

- 制御対象データが設定時間以上古い場合、IR送信されない。
- 古いデータであることをログと状態APIから確認できる。
- 新しいデータ取得後の復帰条件が定義されている。

#### 注意点

raspi4B自身の制御用温度を取得できている間は、今すぐの最優先課題とはしない。
保留解除時に、制御対象の識別方法と鮮度閾値を再確認する。

### DATA-001: `raspi0-1.local`の名前解決失敗を整理する

- **ID:** DATA-001
- **状態:** 保留
- **優先度:** P2

#### 背景

`raspi0-1.local`の名前解決または接続に失敗し、古いデータが残っている。

#### 目的

不要な監視対象と必要な監視対象を整理し、接続失敗ログと古い表示データを減らす。

#### 方針案

- 不要な端末であれば監視対象から外す。
- 必要な端末であればhostname、固定IP、mDNS、Avahi設定を確認する。
- 接続失敗ログの出力頻度を制限する。
- UI上で到達不能または古いデータと分かる表示を検討する。

#### 完了条件

- `raspi0-1.local`を監視対象として残すかどうか決定している。
- 残す場合は安定して名前解決またはIP接続できる。
- 外す場合は不要な取得処理と接続失敗ログが発生しない。

#### 注意点

### DB-003: DB肥大化とデータ保持方法を整備する

- **ID:** DB-003
- **状態:** 未着手
- **優先度:** P3

#### 背景

温湿度、CO2、エアコン状態を同一DBへ保存し続けるため、長期運用でDBが肥大化する。

#### 目的

必要なデータを保持しながら、本番DBのサイズと保守時間を管理可能にする。

#### 方針案

- 本番DBのデータ保持期間を決める。
- 古いデータをアーカイブDBへ移動する。
- 集計後に生データを削除する案を検討する。
- リプレイテスト用DBを本番DBから分離する。

#### 完了条件

- 保持期間、アーカイブ、削除、復元の方針が決まっている。
- 本番処理を長時間止めずに保守できる。
- リプレイテストが本番DBを変更せず実行できる。

#### 注意点

バックアップ方式は`DB-002`で先に確立する。
